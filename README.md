# 策划变更审阅工作区

把会议纪要或零散灵感视为对现有策划体系的一组增量变更，先完成证据对齐、冲突检查和影响面审阅，再跟踪结论是否真正进入策划案。

## 核心流程

1. 在“知识库”指定本地策划文件夹，并把资料标记为“核心”“参考”或“忽略”。
2. 在“审阅”粘贴会议纪要或灵感，填写当前目标。
3. 系统区分决策、提案、问题、行动、风险和信息。
4. 每个讨论点只引用检索到的真实知识片段，并保留文件路径与行号。
5. 查看新增、修改、冲突、缺口，以及直接影响、下游影响和相关讨论点。
6. 人工选择“纳入变更”“需澄清”或“暂不纳入”。
7. 审阅完成后生成变更包，记录每条会议决定的预期结果；首次使用版本流程时人工确认当前工作区为正式基线。
8. 手动修改本地策划案后扫描工作区变化，系统将确定性差异块自动关联会议结论，无法关联的变化保持“未归属”。
9. 对每个差异块选择“纳入本版”“暂时搁置”或“不纳入”，必要时拆分差异块，再生成不修改源文件的候选正式版。
10. 候选验证比较上一正式版、会议决定与候选正式版；逐条确认“已落实”“保持不纳入”“仍需澄清”“意外写入”等结果。
11. 口径冲突、旧口径残留、未归属变化和待确认结果全部处理后，使用预览哈希与精确确认文字发布不可变正式版。
12. 已完成结论随正式版关闭；“需澄清”保留重新决策入口。旧 V1/V2/V3 验证记录继续保留。

## 启动

```powershell
npm install
npm run dev
```

自动化验证：

```powershell
npm test
```

前端默认地址：`http://127.0.0.1:5173`
接口默认地址：`http://127.0.0.1:8787`

## 数据

- 模型设置保存在项目根目录的 `.env`。
- 会议、审阅项和变更包保存在 `data/store.json`。
- 旧流程的修改前版本和 V1/V2/V3 验证版本继续保存在 `data/snapshots`，内容按 SHA-256 去重。
- 工作区修订和检查点保存在 `data/version-archive` 的独立 Git 镜像中；源知识库始终只读。
- 正式版本清单、候选口径、冲突、采纳决定及会议追溯关系保存在 `data/store.json`；`canonicalHeadId` 是新会议默认读取的知识权威。
- 上传文件保存在 `data/uploads`。
- 手动导出的 Markdown 保存在 `data/outputs`。
- 本地知识库始终只读扫描，不会自动修改源文件。

未配置模型时会使用保守的确定性规则；配置 OpenAI 兼容接口后，模型参与讨论点提取、语义分组、候选冲突判断和会议结果验证。确定性补丁、发布门禁与最终人工确认不交给模型替代。
