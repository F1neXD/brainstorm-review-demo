# 归档引擎选型记录

## 结论

版本管理主引擎采用独立 Git 镜像。镜像只复制读取源策划工作区，候选版本、恢复结果和 Git 元数据全部位于工具管理的数据目录，任何归档操作都不写回源文件。

现有 SHA-256 快照在迁移完成前继续作为会议验证的安全备份，但不与 Git 镜像共同承担“正式版本”权威。正式版本能力接入后，Git 修订和 `canonRelease` 是唯一发布依据。

## 已验证能力

- Windows 中文目录与中文文件名。
- 文件新增、修改、删除和 Git 改名检测。
- 二进制文件完整归档与恢复。
- 只修改 `mtime` 不产生修订。
- 从基线只应用选定补丁，生成独立候选修订。
- 正式版本引用不可覆盖。
- 从任意历史修订恢复到工具管理目录。
- 归档、候选和恢复过程不修改源工作区。

## 降级和失败边界

下列情况进入 `snapshot-fallback`：

- 系统未安装 Git 或 Git 不在 `PATH`。
- Git 镜像初始化失败，且修复后仍无法通过完整性检查。

降级模式继续使用现有内容寻址快照完成会议前后比对，但禁用部分补丁采纳、正式发布和历史恢复，避免同时存在两个正式真相源。界面必须明确显示降级状态；不能静默把旧快照当作正式基线。

下列情况不自动降级，而是阻止操作并要求修复：

- Git 对象或正式版本引用损坏。
- 候选补丁无法确定性应用。
- 归档目录与源工作区互相包含。
- 发现符号链接或路径越界。

## 恢复策略

- 代码恢复点：GitHub `main` 上每个实施阶段的独立提交。
- 真实数据备份：项目外时间戳目录，包含完整 `data`、`.env`、源码 ZIP 和 SHA-256 清单。
- 版本内容恢复：只恢复到归档引擎的 `restores` 目录；第一版不覆盖真实策划文件。
