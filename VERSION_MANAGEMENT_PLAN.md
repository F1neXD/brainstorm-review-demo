# 策划版本管理改造计划

## 0. 文档状态

- 当前阶段：阶段 0 已完成，准备实施阶段 1。
- 当前系统：会议审阅、知识库扫描、落实清单、修改前快照、多轮变更验证已可用。
- 当前数据：真实会议与配置必须原样保留；策划源文件继续只读。
- 计划目标：把现有工具升级为“工作区修改 -> 自动归档 -> 选择性采纳 -> 口径收敛 -> 发布正式基线 -> 下一轮继续比较”的本地策划版本工作区。

## 1. 核心规则

### 1.1 三类版本必须分开

1. 工作区：本地策划文件夹中的当前内容，允许存在错误、草稿和冲突。
2. 自动存档：每次真实内容变化产生的历史修订，不代表已经采纳。
3. 正式基线：经过人工选择和发布门禁后形成的不可变版本。

下一轮默认比较：

```text
当前工作区 vs 最新已发布正式基线
```

不能比较“当前工作区 vs 时间最新的自动存档”。

### 1.2 修改时间的职责

- `mtime` 只用于发现文件可能发生变化和辅助展示最近活动。
- 内容哈希决定是否产生新修订。
- 人工发布记录决定哪个版本正式生效。
- 文件名版本号只作为显示和迁移线索，不作为最终权威。

### 1.3 两条独立状态轴

资料权重：

```text
核心 / 参考 / 忽略
```

版本状态：

```text
当前正式 / 工作草稿 / 历史版本 / 待归类
```

两者不能复用同一个字段。

### 1.4 “口径收敛”的可执行定义

系统不能保证策划内容绝对正确，只能保证正式发布前满足：

- 所有检测到的变化都有归属。
- 所有变化都被选择为采纳、搁置或不纳入。
- 没有未处理的已检测冲突。
- 同一文档族只有一个正式生效版本。
- 被替代口径已删除、失效或转入历史。
- 每个采纳结论都有修改后证据。
- 发布动作经过人工最终确认。

## 2. 技术总原则

### 2.1 不直接修改策划源文件

第一版版本管理只读取工作区。采纳、不纳入和发布作用于工具管理的候选版本与正式基线，不自动覆盖用户文件。

自动写回源文件必须作为后续独立能力，并需要明确确认、备份和回滚。

### 2.2 模型不负责发现文件变化

框架负责：

- 文件扫描。
- 内容哈希。
- 新增、删除、重命名和行级差异。
- 候选片段召回。
- 版本权威过滤。

模型只负责：

- 将差异整理成语义变更。
- 判断冲突、替代关系和影响面。
- 为人工审阅提供建议。

### 2.3 归档引擎先做技术验证

首选方案是工具内部维护一个隐藏的 Git 镜像仓库，用户不需要操作 Git。现有 SHA-256 快照机制在迁移完成前继续作为安全备份。

正式选型前必须验证：

- Windows 中文路径。
- 文件新增、删除、重命名。
- 单文件部分差异采纳。
- 从任意历史版本恢复。
- 二进制附件归档。
- 未安装 Git 时的降级路径。

如果内部 Git 镜像无法稳定通过验证，则继续使用现有内容寻址对象库，并在其上实现清单和补丁层。不能同时维护两个正式版本真相源。

### 2.4 代码实施本身需要恢复点

当前项目目录不是 Git 仓库。正式开工前必须二选一：

1. 推荐：初始化独立的代码 Git 仓库，排除 `.env`、真实数据和快照对象。
2. 备选：每个阶段生成完整源码备份和改动清单。

没有代码恢复点时，不进入数据结构迁移。

## 3. 里程碑

| 里程碑 | 包含阶段 | 可交付结果 |
| --- | --- | --- |
| A：可靠归档 | 0-3 | 自动保存修订、正式基线、历史恢复、版本权威 |
| B：选择性采纳 | 4-5 | 行级差异、语义变更、采纳/搁置/不纳入、候选版本 |
| C：口径发布 | 6-7 | 冲突检查、发布门禁、会议到正式基线闭环 |
| D：完整产品 | 8-9 | 完整版本界面、故障恢复、附件归档和最终验收 |

里程碑 B 是版本管理 MVP；里程碑 C 是业务闭环 MVP。

每个变更单元支持：


任务：

- [ ] 根据已采纳补丁生成候选正式版，不修改源工作区。
- [ ] 搁置内容进入待处理架，不参与当前发布。
- [ ] 不纳入内容保留决策记录；内容未变化时不重复打扰。
- [ ] 被拒绝补丁发生变化后重新进入审阅。
- [ ] 支持预览候选正式版完整文件。

验收门禁：

- 候选正式版只包含已采纳变化。
- 拒绝或搁置内容不会进入下一正式基线。
- 任意采纳操作均可撤销到发布前状态。

## 阶段 6：口径索引、冲突检查与发布门禁

状态：`未开始`

依赖：阶段 5。

目标：正式发布前让有效口径收敛。

任务：

- [ ] 从候选正式版抽取规范化策划口径及精确来源。
- [ ] 记录口径的新增、修改、替代和失效关系。
- [ ] 检查同一规则在多文档中的互斥表述。
- [ ] 检查已采纳新口径是否仍有旧口径残留。
- [ ] 检查未归属变化、待审变化和多个正式版本。
- [ ] 对模型无法判断的冲突强制人工确认。
- [ ] 实现发布门禁和门禁失败清单。
- [ ] 发布后生成不可变版本号、清单、说明和内容哈希。

发布门禁：

- 未归属变化为 0。
- 待审变更为 0。
- 未解决冲突为 0。
- 每个文档族最多一个正式修订。
- 所有采纳项均有修改后证据。
- 用户完成最终发布确认。

验收门禁：

- 不满足发布条件时无法更新 `canonicalHead`。
- 发布后仍能完整恢复上一个正式基线。
- 发布失败不会生成半成品正式版。

## 5. 中断恢复协议

每次实施结束前必须更新本节，不允许只在聊天中记录进度。

```text
当前执行阶段：阶段 1 待开始
最后完成任务：阶段 0 / 前置保护、归档引擎和恢复验证
下一个任务：阶段 1 / schema v4、幂等迁移与失败回滚
最近测试结果：npm test 2/2 通过；npm run build 通过
数据 schema：当前代码兼容 v3，真实 store 仍可能是 v2
已知阻塞：无
```

每个阶段结束时必须：

1. 更新阶段状态和复选框。
2. 记录修改文件和数据迁移。
3. 写明测试命令及结果。
4. 记录未解决问题。
5. 生成可恢复检查点。
6. 下一次开始时先读取本文件，不从聊天记忆猜测进度。

## 6. 禁止事项

- 不用修改时间自动决定正式版本。
- 不让模型直接改写正式策划文件。
- 不把自动存档等同于正式发布。
- 不在存在未归属变化时发布。
- 不因模型低置信度而静默忽略变化。
- 不覆盖真实 `store.json` 后再补迁移方案。
- 不在没有恢复点的情况下连续跨阶段修改。
